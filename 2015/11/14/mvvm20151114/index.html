Blend SDK MVVM实现原理分析前篇

发表于 2015-11-14

阅读以下内容你需要理解依赖对象、依赖属性、XAML的基本原理和XAML的附加属性。

1、什么是MVVM

Windows Phone Silverlight客户端一般由XAML、代码程序集和资源组成，这种结构和Android有类似的地方，但与Android不同的是，Android所使用的XML仅仅只是用于描述界面信息，而Windows Phone所使用的XAML具有强大的类型系统描述功能和运行时语义定义功能，这些功能赋予了我们良好运行时脚本编写能力，可以在运行时动态绑定类型和方法，这就为我们创建低耦合程序创造了便利的条件。在Blend Toolkit中，微软提供了2种设计模式用来扩展Silverlight元素的行为，分别基于Behavior行为和Trigger触发器，几乎每一个介绍这种扩展模型的作者，无不流露出对其设计的赞赏。
而MVVM是由这种扩展模型发展而来的专为WPF打造的模式, MVVM是MVC的一个变种, 它的目标是从视图层移除几乎所有代码隐藏（code-behind）。交互设计师可以专注于使用XAML表达用户体验需求，然后创建和视图模型的绑定，而视图模型则是由应用程序开发者开发和维护的。 就像其他架构模式一样，其目的则是程序的扩展性、可维护性和复用性。

2、一切的基石

如果你没有使用”数据绑定”的习惯, 那么研究MVVM就没有多大意义，因为数据绑定是一切的基石。而在WPF中数据绑定都是由依赖属性系统完成的。
WPF中依赖属性是由WPF系统维护的属性系统，它贯穿于整个应用体系中的，用来实现样式，自动数据绑定，动画等等，以至于后面你恨不得所有的属性都是依赖属性。数据绑定和依赖属性它们都是MVVM中的重要组成概念。
在MVVM实现中还有一个重要概念被运用在XAML中，称之为附加属性，这种被称为附加属性的属性元素可以被附加到任何元素节点中，运行时环境通过解释这种附加行为来实现巧妙的代码设计机制。

如果您的类将附加属性严格定义为用于其他类型，那么该类不必从 DependencyObject 派生。 但是，如果您遵循使附加属性同时也是一个依赖项属性的整体 WPF 模型，则需要从 DependencyObject 派生。
通过声明一个 DependencyProperty 类型的 public static readonly 字段将附加属性定义为一个依赖项属性。 通过使用 RegisterAttached 方法的返回值来定义此字段。 为了遵循命名标识字段及其所表示的属性的已建立 WPF 模式，字段名必须与附加属性名一致，并附加字符串Property。 附加属性提供程序还必须提供静态的 Get属性名 和 Set属性名 方法作为附加属性访问器，否则会导致属性系统无法使用您的附加属性。

4、MVVMLight中的实现和优缺点

如果你弄清楚了上文所描述代码调用关系，那么你就能很容易理解此模式实现的关键，在MVVMLight库中，其EventToCommand正是继承了TriggerAction类型，它只是简单的用WPF中的Command模式扩展了Blend SDK的工作机制。而这种基于附加属性的扩展方式正是几乎所有的MVVM库的核心工作方式。
从实现方式上可以看出，MVVMLight优点明显，彻底解耦整个系统，界面不必关心具体业务方式，使整个项目可以并行执行，即美工作美工，程序只要最后将业务绑定至界面即可。
缺点也很明显，从代码上看，微软的官方实现最大的弊端是强事件模式，当源和处理者生存周期不一致时委托部分将会造成内存泄露，影响应用程序性能和稳定性。
